## 1. Prerequisites

### On your VPS:
- **OS:** Ubuntu 22.04+ / Debian 12+ / any Linux with Docker support
- **RAM:** 512MB minimum (1GB recommended)
- **Disk:** 1GB free space minimum
- **Docker Engine:** version 24+
- **Docker Compose:** v2 (the `docker compose` plugin, not the old `docker-compose`)

### On your local machine:
- SSH access to your VPS
- The project folder (this directory)
- `scp`, `rsync`, or `git` for file transfer

### Install Docker on VPS (if not installed):

```bash
# SSH into your VPS
ssh user@YOUR_VPS_IP

# Install Docker (official method)
curl -fsSL https://get.docker.com | sh

# Add your user to docker group (avoids needing sudo)
sudo usermod -aG docker $USER

# Log out and back in for group change to take effect
exit
ssh user@YOUR_VPS_IP

# Verify
docker --version
docker compose version
```

## 5. Build and Deploy

```bash
# Build all containers and start in detached mode
docker compose up -d --build
```

This will:
1. Build the Python backend image (installs dependencies, copies code)
2. Build the frontend nginx image (copies static files)
3. Pull the nginx:alpine image for the gateway
4. Create a named volume `db_data` for persistent SQLite storage
5. Start all 3 containers

**First build takes 1-3 minutes** (downloading base images + pip install).  
Subsequent builds take 10-30 seconds (Docker layer caching).

## 7. Set Up Domain + HTTPS

### Step 1: DNS Configuration

Add an A record in your domain's DNS settings:

```
Type: A
Name: @ (or subdomain like "links")
Value: YOUR_VPS_IP
TTL: 300
```

Wait for DNS propagation (usually 1-15 minutes).

### Step 2: Install a reverse proxy with SSL

Choose ONE of the following methods:

---

### Method A: Caddy (easiest — auto HTTPS)

```bash
# Install Caddy
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update
sudo apt install caddy

# Configure
sudo nano /etc/caddy/Caddyfile
```

Caddyfile content:

```
yourdomain.com {
    reverse_proxy localhost:8098
}
```

```bash
# Reload Caddy (auto-obtains SSL cert)
sudo systemctl reload caddy
```

Done. Caddy automatically gets and renews Let's Encrypt certificates.

---

### Method B: Nginx + Certbot (traditional)

```bash
# Install nginx and certbot
sudo apt install nginx certbot python3-certbot-nginx -y

# Create nginx config
sudo nano /etc/nginx/sites-available/ironylabs
```

Nginx config:

```nginx
server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;

    location / {
        proxy_pass http://127.0.0.1:8098;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 60s;
        proxy_send_timeout 60s;
    }
}
```

```bash
# Enable the site
sudo ln -s /etc/nginx/sites-available/ironylabs /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

# Get SSL certificate
sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com

# Auto-renewal is set up automatically
sudo certbot renew --dry-run
```

## 8. Firewall Configuration

### UFW (Ubuntu/Debian):

```bash
# Allow SSH (don't lock yourself out!)
sudo ufw allow 22/tcp

# Allow HTTP and HTTPS (for your reverse proxy)
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# Allow port 8098 only if NOT using a reverse proxy
# (skip this if using Caddy/Nginx above)
sudo ufw allow 8098/tcp

# Enable firewall
sudo ufw enable
sudo ufw status
```

### If using a reverse proxy (Caddy/Nginx):

You do NOT need port 8098 open to the public. The reverse proxy handles external traffic on 80/443 and proxies internally to 8098.

```bash
# Only allow 8098 from localhost (more secure)
sudo ufw deny 8098/tcp
```


```bash

# Copy backup into the container
```

## 10. Updates and Redeployment

### When you have code changes:

```bash
# Transfer updated files to VPS (rsync recommended)
rsync -avz --exclude=".env" --exclude="__pycache__" ./  user@VPS:~/apps/ironylabs-links/

# SSH into VPS and rebuild
ssh user@VPS
cd ~/apps/ironylabs-links
docker compose up -d --build
```

### Data safety during updates:

- The SQLite database lives in a Docker named volume (`db_data`)
- `docker compose up --build` preserves volumes — your data is safe
- Only `docker compose down -v` destroys the volume (never run this unless intentional)

### Zero-downtime update (advanced):

```bash
# Build new images without stopping
docker compose build

# Restart one by one
docker compose up -d --no-deps backend
docker compose up -d --no-deps frontend
# Gateway doesn't need rebuild (it's nginx:alpine from Docker Hub)
```

## 12. Troubleshooting

### Container won't start / keeps restarting:

```bash
docker compose logs backend --tail 30
```

Common issues:
- **bcrypt error:** Ensure `bcrypt==4.0.1` is in `backend/requirements.txt`
- **Permission error on /app/data:** The volume mount should handle this, but if not: `docker compose down && docker compose up -d`
- **Port conflict:** Another service using 8098: `sudo lsof -i :8098`

### Can't reach the site externally:

1. Verify containers are up: `docker compose ps`
2. Test locally: `curl http://localhost:8098/api/health`
3. Check firewall: `sudo ufw status`
4. Check if port is listening: `sudo ss -tlnp | grep 8098`

### Admin login fails:

1. Verify `.env` has correct password
2. Restart backend to pick up env changes: `docker compose restart backend`
3. Check logs: `docker compose logs backend | grep -i error`

### Database is empty after rebuild:

- Data lives in the `db_data` volume, not the container filesystem
- If you ran `docker compose down -v`, the volume was deleted
- Restore from backup if available

### SSL certificate issues:

```bash
# Test cert renewal
sudo certbot renew --dry-run

# Force renewal
sudo certbot renew --force-renewal
```

## 13. Security Hardening

### Production checklist:

- [ ] Change admin password from default (123456) to a strong one
- [ ] Set a random 32+ character SECRET_KEY
- [ ] Use HTTPS (Caddy or Certbot)
- [ ] Close port 8098 externally (use reverse proxy)
- [ ] Keep Docker and OS updated
- [ ] Set up automated backups
- [ ] Don't expose Docker socket to the internet

### Rate limiting (optional, in your external Nginx):

```nginx
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

server {
    ...
    location /api/ {
        limit_req zone=api burst=20 nodelay;
        proxy_pass http://127.0.0.1:8098;
        ...
    }
}
```

### Fail2ban for admin login (optional):

```bash
sudo apt install fail2ban
# Configure a jail for repeated 401 responses on /api/auth/token
```
